## What I Actually Understand Now

- `Harness Engineering` is a stronger lens than plain tool use because it explains the full workbench around an agent: task environment, action surfaces, extension surfaces, automation, governance, and feedback loops.
- AI memory should not be collapsed into one bucket; the most useful distinction is now between weights, chat/project memory, runtime state, durable memory, artifacts, and background consolidation.
- `OpenClaw` is especially valuable because it makes durable memory explicit and file-based, which makes layered memory design much easier to reason about.
- The `Self-Improving-Agent` skill is a very useful concrete sample because it turns self-improvement into a bounded workflow: log learnings, promote durable rules, and only then extract reusable skills.
- The new missing piece is runtime realism: source plugin path, live cache, installed-path symlinks, and real hook payloads are not side details; they are part of the engineering problem.
- The newer progressive line is useful because it stops mixing layers: prompt is not context, tool use is not the same as CLI/MCP, and skills/plugins are not the same thing as harness.
- The newest refinement is that the line is now split into teachable subdomains instead of one long ladder: inner control, action surfaces, extension surfaces, harness, and runtime governance.
- We then extended that branch with the missing risk layer: memory poisoning, shared-memory boundary design, incident libraries, eval gates, release gates, and rollback discipline.
- `OpenClaw` is most useful to read as a self-hosted assistant runtime with a very explicit `skills + plugins + apps + automation` operating layer.
- `Codex`, `Claude Code`, `Gemini CLI`, `Grok`, and `Kimi` are not just “different models”; they represent different bets on entry surface, action surface, and extension surface.
- `LLMOps` platform choice is really about choosing a control point: lifecycle governance, production observability, pre-release eval, or app workbench.
- `AI Security` is not the same as `AI Safety`; it is more about attack surfaces, trust boundaries, artifact integrity, and runtime controls.
- A strong AI vault needs at least three entry surfaces: topology (`AI 总控制塔`), question routing (`AI 问题导航`), and choice/judgment (`AI 决策导航`).
- A stronger vault also needs an evidence-choice layer: when to look at people, organizations, systems, or cases, because these four assets answer different questions.
- The next maturity step after evidence-choice is method: how to read people for route/divergence signals, and how to read cases for rollout packets.
- A still stronger step is method completeness: not just people and rollout, but also organizations and failures, because expertise depends as much on reading capability/adoptability and failure boundaries as on reading success.
- The new paper refinement is that recent AI papers should not be stored as a list; the useful unit is now `paper -> model route -> system route -> product route`.
- The recent-five-year line is especially useful because it explains why modern AI changed shape through `post-training`, `reasoning`, `tool use`, `open-weight`, and `reasoning by RL`, not just through bigger pretraining.
- The newest refinement is that 2024-2026 needs a second recent-paper layer: not just mainline papers, but route supplements for `long context`, `multimodal productization`, and `computer use`.
- The newest refinement after that is that multimodal should not stay bundled: `voice / realtime` and `OCR / document AI` each need their own `paper -> topic -> system -> product` route layer.
- The newest refinement after that is that recent model refresh should also be its own recall layer: not just papers and routes, but `what changed this quarter in model land`.
- The newest refinement after that is priority: recent AI should not be stored as a flat feed; the strongest lens is now to rank which lines deserve deep time, especially `Harness / Eval / Reasoning+Coding+Computer Use`.
- The newest refinement after that is resource allocation: a strong expert vault still reads the edges, but study time should be layered, with heavy investment on the main trunks and lighter ongoing coverage on the edges until they prove they belong in the main line.
- The newest refinement now includes expression readiness: an expert vault should not only store knowledge, but also provide interview-grade answer structures that connect architecture choices, risk boundaries, and production controls.
- The newest refinement after that is that expression readiness is still not enough if the pack starts from technical questions; a more realistic senior interview flow starts from self-positioning, representative projects, and owner boundaries before the technical deep dives.
- The newest refinement after that is that senior signal often appears in cross-functional language: why not to launch, how to align product/platform/security, and how to turn one project into a team mechanism.
- The newest refinement after that is that “knowledge breadth” in interviews is not just more model facts; it also includes open-ended system design, capacity and cost reasoning, and real incident handling.
- The newest refinement after that is that breadth alone is still not enough; a stronger interview system must also translate those abstractions into company-shaped scenarios that sound like real target roles.
- The newest refinement after that is that broad AI study also needs a domain skeleton: each point should belong to a trunk, each trunk to a domain plane, and each plane to a clear area layer.

## What Still Feels Fuzzy

- We now have a much better harness map, but we still need more grounded case studies about release gates, plugin ecosystems at scale, and real automation failures.
- The new AI memory line is coherent, and now also has a self-improving skill spine plus a real governance layer. The next judgment call is whether it deserves its own top-level area or should stay inside AI-Learning + AI-Engineering as a cross-cutting specialty.
- We now also have a Mac-first local lab path, but we still need to actually execute it to turn structure into skill.
- AI security now has better control points and release-gate language, but it can still use more real incident and org-process depth later.
- The recent-paper layer is still thinner than it should be on 2024-2026 long-context, multimodal, computer-use, and system-card material, so the newest route map is strong but not yet fully exhaustive.
- The newest missing edges are no longer the route skeletons themselves, but the next layer after them: benchmark / eval / system-card depth, and clearer operating boundaries for production deployment.
- The newest missing edge after the model refresh is now clearer benchmark / eval / pricing-tier / deployment-fit material, so the newest models can be judged inside the vault rather than from launch prose alone.
- The newest gap after the priority layer is depth imbalance: the top three lines are now clear, but they still need much thicker case studies and operating packets than the lower-priority branches.
- The newest leverage point is no longer adding random AI notes, but thickening the new main trunks with drill-down pages, especially for model formation, capability upgrading, and runtime engineering.
