# 恢复笔记

## Last Session

- We pushed the MLOps / LLMOps branch from lifecycle basics into enterprise platform tradeoffs, then expanded the AI security branch into a more usable control stack with practical cases, release gates, reusable templates, and an explicit enterprise operating model.
- We then returned to the agent engineering branch and reorganized it around `Harness Engineering`, vendor action surfaces, extension surfaces, and automation planes.
- This round extended that branch into workflow modes, automation loops, case comparisons, and a minimal `Harness Lab` project pack.

## What I Actually Understand Now

- `Harness Engineering` is strongest when read as a workbench model, not just a tool-calling story.
- `Terminal`, `desktop / browser`, `cloud`, and `channel` are four distinct workflow modes, and each favors different vendor systems.
- `OpenClaw`, `Codex`, `Claude Code`, `Gemini CLI`, and `Grok` are easier to compare by entry surface, extension surface, and automation surface than by raw model rhetoric.
- The automation plane is one of the real differentiators: hooks, cron, CI, background tasks, and channel events create very different operational behaviors.
- A usable harness learning path now needs both concept notes and practical experiments.

## What Still Feels Fuzzy

- The branch is now structured and more practical, but it still needs more real delivery cases around release gates, plugin ecosystems, and automation failures at scale.
- We still lack richer incident libraries that connect harness design to security and rollout mistakes.
- The new `Harness Lab` is a strong practice entry, but it still needs real hands-on runs later to become experience.
